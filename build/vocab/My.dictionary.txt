LIT||/* Diatonic Checking Information
LIT||* In order to determine whether a chord is diatonic to a key, the analysis
LIT||* requires knowledge of two things: one, the equivalences between possible chord
LIT||* qualities (such as M7 and M9) and two, the base chords which are diatonic to
LIT||* the key of C for a given mode.
LIT||*
LIT||* Equivalence rules have the format
LIT||*     (equiv [basic chord] [equivalent chord] [equivalent chord] ...)
LIT||* where the basic chord is one that might arise in the actual diatonic mapping of
LIT||* a key below, and the equivalent chords are chords of the same root with
LIT||* different qualities.
LIT||*
LIT||* Diatonic rules have the format
LIT||*     (diatonic [mode] [list of all chords fitting the key of Cmode] )
LIT||* with the mode generally being capitalized for formatting purposes in the
LIT||* leadsheet and roadmap modes.
LIT||*/

LIT||/* Brick Definitions
LIT||* A chord brick can be comprised of anything from a single chord to a series of
LIT||* complex bricks.
LIT||*
LIT||* Each brick is defined by five things: its name (with any qualifier information
LIT||* needed), its mode, its type, its key, and the list of subblocks composing it.
LIT||* A brick definition thus has the following format:
LIT||*
LIT||*     (defbrick [name-spaced-by-hyphens]([qualifier]) [mode] [type] [key]
LIT||*             ([first subblock])
LIT||*             ([second subblock])
LIT||*             ...
LIT||*             ([last subblock]))
LIT||*
LIT||*  Name: Each brick must have a name by which it can be identified in the
LIT||*        brick library. If a name has multiple words, then one may replace the
LIT||*        spaces in the name with hyphens so as to keep the name in one unit.
LIT||*        Additionally, if a user wishes to have multiple different brick
LIT||*        definitions for the same brick, she may add a qualifier to the name
LIT||*        of each brick in parentheses directly after the name. Two bricks may
LIT||*        have the same name, and will be treated as the same brick in parsing,
LIT||*        but no two may have the same name-qualifier pair. The first definition
LIT||*        of a brick with a given name will be taken as the default.
LIT||*
LIT||*  Mode: A brick generally has a single mode which describes the chord sequence
LIT||*        it contains. Generally, this will either be "Major" or "Minor", based
LIT||*        upon whether the brick has a "straight" or a "sad" feel or what the
LIT||*        mode is of the "on" chords of the brick.
LIT||*
LIT||*  Type: A brick has a type based upon the purpose it serves; for instance, a
LIT||*        Cadence introduces a resolving musical idea. A given brick type must
LIT||*        be listed at the top of the file in order to be treated as a valid
LIT||*        type in the roadmap. Brick-type definitions are formatted as:
LIT||*
LIT||*            (brick-type [type name]  [cost])
LIT||*
LIT||*        where cost specifies how much it is favored in parsing. The larger the
LIT||*        cost number, the less preferred a brick is as explanation for a series
LIT||*        of chords when a piece is analyzed.
LIT||*
LIT||*        A special type, Invisible, accounts for bricks which will not be shown
LIT||*        to the user in parsing but might help generalize pieces of complex
LIT||*        bricks with many variations. If a brick is specified as "Invisible",
LIT||*        it will not be shown to the user at all in Impro-Visor, but any brick
LIT||*        definition depending on the brick will still be able to function as
LIT||*        it normally would.
LIT||*
LIT||*  Key:  The predominant key of the brick should be based upon where the brick
LIT||*        does resolve, or in the case of an unresolved sequence where it would
LIT||*        resolve after the sequence. Usually, the key specified in this library
LIT||*        is C; however, this is not a prerequisite for a definition.
LIT||*
LIT||*  Sub-blocks: A "block" is either a brick or a chord; so, a "sub-block" is a
LIT||*        brick or chord that acts as part of a definition. These are listed
LIT||*        after the basic brick information in order of occurrence in the brick.
LIT||*        If a sub-block is a chord, then it is written as
LIT||*
LIT||*            (chord [chord name] [relative duration])
LIT||*
LIT||*        whereas a brick sub-block would be listed as
LIT||*
LIT||*            (brick [brick name] [key] [relative duration])
LIT||*
LIT||*        to appropriately describe the default version of that brick. If a sub-
LIT||*        brick is supposed to be by default a variation on a given brick's
LIT||*        definition rather than the original, then the quality can be specified
LIT||*        in the definition as
LIT||*
LIT||*            (brick [brick name]([qualifier]) [key] [relative duration])
LIT||*
LIT||*        to make the default in the dictionary appear with that variation as a
LIT||*        sub-block. However, when parsing, the parser will treat all bricks
LIT||*        with the same name as substitutable for that brick, so this usage will
LIT||*        not allow a brick to have a more specific definition in all cases.
LIT||*
LIT||*        Relative durations describe the ratio of the durations of each sub-
LIT||*        block to the others in a given brick. These relative durations should
LIT||*        have values no less than the number of chords composing a given sub-
LIT||*        block. These are used only for the specification of bricks in the user
LIT||*        library, as analysis ignores chord durations.
LIT||*
LIT||* With this information, a brick will be defined in the BrickLibrary and used in
LIT||* analysis as a possible parse for a given brick. Bricks may be added either by
LIT||* hand in this file or, more easily, by generating the brick in Impro-Visor's
LIT||* Roadmap mode, selecting it, and clicking the "New Brick" button. The definition
LIT||* will then be added to the bottom of this file to be loaded in future Impro-Visor
LIT||* use.
LIT||*
LIT||* If you introduce a Cadence into the brick dictionary, you will notice that
LIT||* matching Overrun and Dropback versions of the brick will also appear for that
LIT||* brick on runtime. At the present, the only way to remove those bricks is to
LIT||* remove the original cadence.
LIT||*
LIT||* WARNING: Deleting a brick from the brick dictionary manually may destroy the
LIT||* definitions of any bricks using the brick you wish to delete as a sub-block. In
LIT||* order to avoid damaging the dictionary, setting the brick type of a brick you
LIT||* wish to ignore to "Invisible" will retain all dependent definitions without
LIT||* revealing the undesired brick at runtime.
LIT||*
LIT||* Stuff on lines beginning with two slashes or between slash star and star slash
LIT||* are "commented out". They are not in effect, but may be revisited.
LIT||*/
